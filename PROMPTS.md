# Промпт-журнал (AI Layout Assistant)

В этом файле задокументированы ключевые системные инструкции, которые использовались для настройки поведения ИИ в проекте.

---

## 1. Основной промпт генерации (v0.4)
**Цель:** Заставить ИИ строго следовать БЭМ-структуре и использовать только стили из эталона пользователя.

**Текст промпта:**
> Ты — эксперт по БЭМ. Трансформируй текст в HTML-структуру, СТРОГО используя паттерны и классы из "Золотого стандарта". 
> 
> ПРАВИЛА СТРУКТУРЫ:
> 1. "ЭТАЛОН HTML" — это ОБЩАЯ ОБЕРТКА (контейнер) для всей страницы/блока. 
> 2. Используй `<section class="ce-section">` как КОРНЕВОЙ элемент всей верстки.
> 3. Внутри корневого элемента ОБЯЗАТЕЛЬНО добавь центрирующую обертку (например, `<div class="ce-container">`).
> 4. КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО использовать любые теги заголовков (H1-H6), кроме разрешенных пользователем.
> 
> ПРАВИЛА CSS:
> 1. КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО дублировать глобальные стили (reset, fonts) из эталона. 
> 2. Пиши ТОЛЬКО уникальные стили для создаваемого блока.
> 
> ПРАВИЛО ТАБЛИЦ: 
> Если данные выглядят как табличные, ОБЯЗАТЕЛЬНО используй тег `<table>` с классами `ce-table`, `ce-table__row`, `ce-table__cell`.

**Что сработало:** 
- Добавление слова "КАТЕГОРИЧЕСКИ" значительно снизило количество галлюцинаций с заголовками.
- Разделение на правила HTML и CSS помогло ИИ не путаться в контексте.

## 2. Промпт для чата правок (Refinement)
**Цель:** Вносить точечные изменения в существующий код, не ломая остальную верстку.

**Текст промпта:**
> Ты — эксперт по БЭМ. Ты должен ИСПРАВИТЬ существующую верстку на основе пожеланий пользователя.
> 
> ИСХОДНЫЕ ДАННЫЕ:
> - Текущий HTML: [CODE]
> - Текущий CSS: [CODE]
> 
> ПРАВИЛА (КРИТИЧЕСКИ ВАЖНО):
> 1. Сохраняй структуру БЭМ и классы из эталонов.
> 2. ЗАПРЕЩЕНО использовать комментарии типа "код остается без изменений". Пиши каждую строчку кода от начала до конца.
> 3. Вноси только точечные правки согласно пожеланию пользователя.

**Что сработало:** 
- Запрет на комментарии типа "..." заставил ИИ всегда возвращать валидный рабочий код, который можно сразу вставлять в iframe.

## 3. Промпт анализа эталона (Reference Parsing)
**Цель:** Разделить монолитный файл верстки на чистый HTML и CSS.

**Текст промпта:**
> Ты — помощник по верстке. Тебе дадут содержимое файла (HTML/CSS). Твоя задача: выделить из него чистый HTML-шаблон и чистый CSS. Верни ответ СТРОГО в формате: ###HTML### [код] ###CSS### [код].

**Что сработало:** 
- Использование жестких маркеров `###` позволило фронтенду надежно парсить ответ регулярными выражениями даже при нестабильном соединении.

---

## 4. Промпт для SEO-микроразметки (JSON-LD)
**Цель:** Генерация валидной схемы Schema.org на основе выбранных пользователем типов.

**Текст промпта:**
> В блоке ###SCHEMA### сгенерируй JSON-LD разметку. 
> ТИПЫ: [Список выбранных схем]. 
> ПРАВИЛА:
> 1. Используй данные из текста пользователя.
> 2. Если данных не хватает — не выдумывай их, оставь поле пустым.
> 3. Верни только чистый JSON.

**Что сработало:** 
- Указание конкретных типов (Article, FAQ) заставило ИИ генерировать разные структуры под разные блоки, а не одну универсальную.

## 5. Борьба с «ленью» ИИ (Text Retention)
**Проблема:** При больших объемах текста ИИ начинал сокращать контент или писать "Текст здесь...", игнорируя реальные данные пользователя.

**Решение (усиление промпта):**
> ОБЯЗАТЕЛЬНО перенеси ВЕСЬ текст пользователя в HTML. 
> НЕ СОКРАЩАЙ, не обобщай и не выбрасывай фрагменты текста. 
> Каждое предложение из входных данных должно найти свое место в верстке.

## Ошибки и итерации (Failed Prompts)

### Попытка №1: "Сделай красиво"
**Результат:** ИИ выдавал случайный дизайн, который полностью игнорировал CSS-стандарты компании.
**Урок:** ИИ нужны жесткие рамки («Золотой стандарт»), иначе он уходит в творчество, которое бесполезно для продакшена.

### Попытка №2: Авто-регенерация при любой смене настроек
**Результат:** ИИ начинал менять весь код, когда пользователь просто хотел поменять тег `<h2>` на `<h3>`.
**Урок:** Мы разделили промпт на "Генерацию с нуля" и "Точечную коррекцию" (Refinement). Теперь ИИ знает, когда нужно творить, а когда — только исправлять детали.

### Попытка №3: Ожидание идеального формата
**Результат:** Когда ИИ не писал маркер `###HTML###`, парсер ломался и выдавал пустой экран.
**Урок:** Промпты — это хорошо, но код должен быть устойчивым. Мы добавили в JS фоллбеки на стандартные блоки кода Markdown (```html).

## Итоги (Lessons Learned)
1. **Меньше — лучше**: Чем короче промпт, тем точнее модель `openrouter/free` (особенно Gemini) ему следует.
2. **Форматирование**: ИИ лучше понимает инструкции, если они разбиты на нумерованные списки и выделены капсом ключевые требования (ОБЯЗАТЕЛЬНО, ЗАПРЕЩЕНО).
3. **Контекст**: Передача "Золотого стандарта" в системном сообщении превращает обычную LLM в специализированный инструмент верстки.
